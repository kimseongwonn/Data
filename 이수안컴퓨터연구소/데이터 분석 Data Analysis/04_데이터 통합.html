데이터 통합

Data Integration

개체 식별 문제(Entity Identification Problem)

  • 데이터 통합(data integration)은 여러 데이터 저장소로부터 온 데이터의 합병
  • 데이터 웨어하우스(data ware house)나 데이터 마이닝(data mining) 같은 데이터 분석 작업은 다수의 원천 데이터로부터 하나의 통일된 데이터 저장소로 결합시키는 통합 작업 필요
  • 데이터 원천은 데이터베이스, 데이터 큐브(data cube), 플랫 파일(flat file) 등 다양한 형태로 존재
  • 여러 데이터 원천들로부터 데이터를 통합할 때, 동일한 의미의 개체들이 서로 다르게 표현되어 있을 경우 어떻게 일치시킬 수 있을까? → 개체 식별 문제(Entity Identification Problem)
  • A, B 두 개의 DB에서 같은 ‘고객’이라는 의미의 table의 이름이 각각 {customer_id : customer} , {cust_number : cust}일 때 즉, 개체의 의미는 같은데 개체의 이름과 key 이름이 다를 때 개체 식별 문제가 발생함
  • 데이터 분석가나 컴퓨터가 개체들이 같은 것인지 식별할 수 있을까?
  • 메타데이터의 활용이 중요
  • 각 속성의 메타데이터는 이름과 의미, 데이터 타입, 허용되는 값의 범위, 무규칙(null rule) 등을 포함
  • 스키마 통합 과정에서 오류를 피하기 위해 메타데이터 활용 필요
  • 메타데이터를 이용해 데이터 변환에서도 사용
    • A 데이터베이스는 성별 값으로 ‘M’, ‘F’ 사용
    • B 데이터베이스는 성별 값으로 ‘1’, ‘2’ 사용
  • A 데이터베이스의 customer 테이블의 customer_id와 B 데이터베이스의 cust 테이블의 cust_number가 기본키(primary key)이고 두 속성 모두 동일한 데이터 타입과 도메인을 가지고 있다면 두 속성은 이름은 다르지만 동일한 속성으로 판단 가능
  • 데이터 통합시 일원화된 속성으로 정의 가능
  • customer 테이블과 cust 테이블 간의 튜플 연계도 가능
  • 데이터베이스의 어떤 속성을 다른 데이터베이스의 속성과 동일성 여부 판단 시 종합적으로 고려할 사항
    • 속성의 데이터 타입과 도메인
    • 기본키 여부
    • 참조 무결성(referential constraint) : 참조하는 테이블의 외래키(foreign key)의 속성 값은 반드시 참조하는 테이블의 기본 키(primary key) 속성 값 중 하나여야 함
    • 함수적 종속 관계(functional dependency)

함수적 종속성(Functional Dependency)

  • 속성 X,Y에 대하여 x1=x2(x1,x2∈X)이면, y1=y2(y1,y2∈Y)임을 보장할 때 속성 Y는 함수적으로 종속됨
  • X→Y로 표기
  • X는 결정자(determinant)
  • Y는 의존자(dependent)
  • X가 Y를 함수적으로 결정
  • Y가 X에 함수적으로 종속
  • 학번에 의해 학생 이름과 학과는 고유하게 구분
  • 학과 속성은 학번에 함수적으로 종속
  • 학번은 결정자, 학생이름과 학과는 종속자
  • 학번→{학생이름, 학과} 로 표기
  • 학번→학생이름
  • {학번,과목코드}→성적
  • {학번,과목코드}→학생이름
  • 학생이름을 결정짓는 요소는 학번
  • 성적을 결정짓는 요소는 학번과 과목코드 (학번 하나 또는 과목코드 하나만으로는 구분 불가능하다 학번과 과목코드를 동시에 고려해야 구분 가능) >> 완전 함수적 종속
  • 학생이름은 학번과 과목코드에 의해서도 고유하게 구분 가능 → 학생이름의 경우 함수적 종속성이 두개 (학생 이름은 학번과 과목코드에 의해서도 고유하게 구분 가능하지만 학번과 과목코드의 일부인 학번으로도 구분이 가능) >> 부분 함수적 종속
  • 추가 공부

    함수적 종속성(Functional Dependency, FD)은 데이터베이스 설계에서 매우 중요한 개념으로, 데이터베이스 테이블의 특정 속성들이 다른 속성들에 의해 고유하게 결정되는 관계를 의미합니다. 함수적 종속성은 데이터베이스의 정규화를 이해하고 수행하는 데 핵심적인 역할을 합니다.

    함수적 종속성의 정의

    함수적 종속성은 다음과 같이 정의할 수 있습니다:

    • 함수적 종속성: 속성 집합 X가 속성 집합 Y를 함수적으로 결정한다고 할 때, X의 값이 동일하면 Y의 값도 항상 동일합니다. 이를 X→Y로 표현합니다.
      • 속성 집합 X의 값이 속성 집합 Y의 값을 함수적으로 결정한다 → X의 속성으로 Y의 속성이 결정된다 즉, Major→Advisor에서 Major가 뭐냐에 따라 Advisor가 결정된다
    • 즉, 테이블에서 X가 같은 두 행은 항상 Y도 같아야 합니다.

    예제

    예를 들어, 학생 정보 데이터베이스에서 다음과 같은 테이블을 고려해봅시다:

    StudentIDStudentNameMajorAdvisor
    1AliceMathDr. Smith
    2BobPhysicsDr. Johnson
    3CharlieMathDr. Smith
    4DavidCSDr. Wang

    여기서 다음과 같은 함수적 종속성을 찾을 수 있습니다:

    • StudentID → StudentName, Major, Advisor : 학생 ID가 같으면 학생 이름, 전공, 지도 교수도 같습니다. 즉, StudentID가 유일한 식별자입니다. (StudentID에 따라서 StudentName, Major, Advisor가 결정이 된다)
    • Major → Advisor : 전공이 같으면 지도 교수도 같습니다. 즉, Major가 Advisor를 결정합니다.

    함수적 종속성의 유형

    1. 완전 함수적 종속성 (Full Functional Dependency):
      • 속성 집합 Y가 X 전체에 함수적으로 종속되지만, X의 어떤 진부분집합에도 종속되지 않을 때 이를 완전 함수적 종속성이라 합니다.
      • 예: StudentID → StudentName (여기서 StudentID 전체가 StudentName을 결정하고, 부분 집합인 StudentID의 일부는 StudentName을 결정하지 않습니다.)
    1. 부분 함수적 종속성 (Partial Functional Dependency):
      • 속성 집합 Y가 X 전체에 종속되지만, X의 진부분집합에도 종속될 때 이를 부분 함수적 종속성이라 합니다.
      • 예: StudentID, CourseID → Grade가 있지만, 실제로는 CourseID → Grade도 성립할 때, 이는 부분 함수적 종속성입니다.
    1. 이행적 함수적 종속성 (Transitive Functional Dependency):
      • X→Y이고 Y→Z일 때, X→Z 를 이행적 함수적 종속성이라 합니다.
      • 예: StudentID → Major이고 Major → Advisor일 때, StudentID → Advisor도 성립합니다.

    함수적 종속성의 활용

    함수적 종속성은 데이터베이스 정규화(normalization)의 기초입니다. 정규화는 데이터 중복을 최소화하고, 데이터베이스의 구조를 개선하여 데이터 무결성을 유지하는 과정입니다. 함수적 종속성은 정규화 과정에서 여러 정규형(normal form)을 정의하는 데 사용됩니다:

    1. 제1 정규형 (1NF): 모든 속성이 원자 값을 가져야 합니다.
    1. 제2 정규형 (2NF): 제1 정규형을 만족하고, 부분 함수적 종속성이 제거되어야 합니다.
    1. 제3 정규형 (3NF): 제2 정규형을 만족하고, 이행적 함수적 종속성이 제거되어야 합니다.
    1. BCNF: 모든 결정자가 후보 키(candidate key)여야 합니다.

    결론

    함수적 종속성은 데이터베이스 설계에서 속성 간의 관계를 명확히 하고, 데이터의 일관성과 무결성을 유지하는 데 중요한 역할을 합니다. 이를 통해 정규화를 수행하고, 효율적이고 중복 없는 데이터베이스 구조를 설계할 수 있습니다. 데이터베이스의 성능을 최적화하고 데이터 무결성을 유지하려면 함수적 종속성에 대한 이해가 필수적입니다.

유도 속성(derived attribute)

  • 어떤 속성은 다른 속성이나 속성의 집합으로부터 유도 가능
    • 연령 속성은 생년월일 속성의 유도 속성
    • 총점 속성은 각 과목 점수 속성 집합의 유도 속성
  • 일반적으로 데이터베이스 조회 성능을 높이기 위해 반정규화 과정에서 유도 속성 정의
  • 반정규화(de-normalization)
    • 정규화(nomalization)의 반대 개념으로 유도 속성의 정의
    • 릴레이션의 수직/수평 분할 등이 존대
    • 일반적으로 정규화는 데이터 중복을 제거하는 과정
    • 반정규화는 성능 향상을 위해 중복을 허용하는 과정
  • 유도 속성은 성능 향상에 도움이 되지만 데이터 중복을 허용하는 과정으로 데이터 불일치 문제 발생
    • 월소득 속성값이 100만원인데 연소득 속성값이 1,000만원 (둘 중 하나의 속성값에 오류가 존재)

정규화되지 않은 테이블

  • 테이블 정규화 : 데이터베이스 내의 테이블 구조를 개선하여 데이터 중복을 최소화하고 데이터의 일관성과 무결성을 유지하고 데이터 구조의 명확성을 높이는 것을 의미
    • 종류 : 제1 정규화(1NF), 제2 정규화(2NF), 제3 정규화(3NF)
  • 정규화되지 않은 테이블의 사용은 데이터 중복의 원인이 됨
  • 일부러 정규화하지 않은 채 테이블을 사용하는 경우, 조인을 피함으로 성능을 향상시키지만 이러한 테이블 구조는 데이터 중복으로 인한 데이터 일관성 저해 문제 발생
  • 예시) 이름과 주소 정보를 키로 사용하여 구매자 주문을 저장하는 데이터베이스
    • 같은 이름에 다른 주소를 갖는 경우데 대해 데이터 불일 치 발생
    • 정규화 관점에서 구매자 주소는 구매자의 구매 품목과는 별개의 테이블에 저장 필요
    • 구매 테이블의 키가(구매자 번호 + 구매 일시)라고 가정할 때, 주소는 키에 부분 함수적 종속 관계에 있으므로 제2정규화 원칙에 따라 구매자 번호화 함께 별도의 테이블로 분리 설계해야함
    • 그러나 조회 성능 향상을 위해 테이블을 분리하지 않았을 때에는 같은 구매자 번호에 대해서 다른 주소가 존재할 가능성은 언제든지 있음

정규화가 필요한 이유

  • 불필요한 데이터 중복으로 공간 낭비와 이상(anomaly) 발생
  • 삽입이상(Insertion Anomaly) : 새 데이터를 삽입하기 위해 불필요한 데이터도 삽입하는 문제
  • 갱신이상(Update Anomaly) : 중복 튜플 중 일부만 변경하여 데이터가 불일치되는 모순의 문제
  • 삭제이상(Delete Anomaly) : 튜플 삭제 시 필요한 데이터까지 함께 삭제되는 데이터 손실

    삽입 이상

    • 아직 수업을 하나도 수강하지 않은 학생의 경우 primary key가 student_id(학번), course_id(과목코드)로 사용되고 있으므로 삽입 불가(학번은 존재하나 과목코드가 없으므로 기본키 컬럼음 NULL 불가) → 삽입이상
    • 삽입 하려면 ‘수강 미정’과 같은 과목코드를 불필요하게 삽입해야함

    갱신 이상

    • ‘이순신’이 ‘기계공학과’에서 ‘컴퓨터공학과’로 전과하면 전체 학과 정보 변경 필요
    • 모두 변경하지 않은 경우(테이블의 일부만 변경할 경우) ‘이순신’이 ‘기계공학과’인지 ‘컴퓨터공학과’인지 알 수 없음(데이터의 불일치)

    삭제 이상

    • ‘김철수’가 ‘1800100’ 과목을 수강취소할 경우 ‘김철수’에 대한 행을 삭제하게 됨
    • 수강취소를 반영하기 위해서 ‘김철수’학생 정보가 전부 삭제되는 이상 발생

정규화(Normal Form)

  • 중복 요소를 찾아 제거해 나가는 과정
  • 제1정규화 : 하나의 속성이 원자값(atomic value)을 갖도록 설계를 변경하는 과정
  • 제2정규화 : 주식별자가 아닌 속성들 중에서 주식별자 전체가 아닌 일부 속성에 종속된 속성을 찾아 제거하는 과정
  • 제3정규화 : 주식별자가 아닌 속성들 중에서 종속관계에 있는 속성을 찾아 제거하는 과정

제1정규화(1NF, First Normal Form)

  • 학번 ‘201814231’과 ‘201859341’의 과목코드와 성적이 원자값을 갖고 있지 않다(여러 속성이 존재한다)

제2정규화(2NF, Second Normal Form)

  • 제1정규화를 거친 테이블에서 부분 함수 종속성을 찾아 제거해준다
  • 그러나 여전히 삽입 이상과 삭제 이상이 남아있다

제3정규화(3NF, Third Normal Form)

  • 학번 → 학과 → 등록금 : 학번 → 등록금이라는 이행적 종속 관계가 존재
  • [학번, 학과] table과 [학과, 등록금] table로 분리해준다
  • 삽입 이상 : 새로운 학과가 생겨도 [학과, 등록금] table에 학과와 등록금만 입력해주면 해결됨
  • 삭제 이상 : ‘201911214’ 학번이 자퇴해서 데이터를 삭제해도 경영학과에 대한 정보는 [학과, 등록금] table에 여전히 남아있으므로 해결됨

상관 분석(Correlation Analysis)

  • 두 속성간의 엄격한 함수적 종속 관계가 성립하지 않지만 상관 분석을 통해 한 속성이 다른 속성을 얼마나 강하게 암시하는지를 데이터를 통해 측정
  • 두 속성 간에 상관도가 높다면 두 속성을 중복으로 보고 그 중 하나의 속성을 제거할 수 있음
  • 두 속성 간의 상관도는 데이터 타입에 따라 다르게 구함
  • 수치 속성에 대해서 속성 A와 B의 상관계수
  • 상관계수의 절대값이 클수록 더 강한 상관성이 있다는 의미
  • 일반적으로 두 속성 산 상관성이 클수록 중복 속성으로 판단해서 하나의 속성을 분석 대상에서 배제해도 분석 결과에 큰 영향을 주지 않음
  • 상관관계가 인과관계를 직접 의미하는 것은 아님(상관 관계가 높다 → 안과 관계가 높다 X 알 수 없다 O)
  • 어떤 지역의 병원 수와 자동차 수 사이에 상관관계를 발견해도 병원 수와 자동차 수 사이의 어떤 인과가 존재한다기 보다는 두 속성 다 인구에 대한 인과관계로 보는 것이 타당
  • 상관분석상 강한 상관관계를 나타낸다고 하더라도 중복 속성 여부를 판단할 때는 해당 분야 도메인 지식을 충분히 고려해서 최종 판단하는 것이 바람직함

카이제곱 검정(Chi-Square Tests)

  • 데이터가 수치형일 때는 상관계수를 구하는 것이 가능하지만 범주형(이산형)일 때는 불가능
  • 범주형(이산형) 데이터인 경우 속성 A와 B사이의 상관관계는 피어슨(Pearson)의 카이 제곱 검정에 의해 측정 가능
  • 속성 A가 c개의 범주 값 a1, a2,…,ac를 가진다고 가정
  • 속성 B가 r개의 범주 값 b1, b2,…,br를 가진다고 가정
  • 속성 A와 B에 의해 구성되는 튜플은 c개의 열과 r개의 행으로 구성되는 분할표로 표현
  • 기대도수 eij 계산식
  • X^2 통계량은 속성 A와 B가 독립이라는 가설을 검증
  • 카이제곱 검정은 자유도(r-1) x (c-1)을 갖는 유의수준에 근거하여 검정

데이터 값 충돌 탐지 및 해결

  • 서로 다른 데이터 원천의 데이터들을 통합할 때 동일한 개체에 대해서도 속성값이 다를 수 있음 → 표현(representation), 척도(scaling), 부호화(encoding) 등의 차이
    • 거리를 나타내는 속성으로 어느 DB에서는 미터(meter)단위로 다른 DB에서는 마일(mile)단위로 저장
    • 학생 성적 데이터로 어느 DB에서는 과목별 점수가 저장되고 다른 DB에서는 총점과 평균만 저장
  • 동일한 개체의 동일한 값이 데이터 원천에 따라 다르게 표현되어 있는 경우, 데이터 통합시에 기준을 정하여 데이터 값을 변환하여 통합시키는 것이 필요
  • 통합 과정에서 DB의 속성을 일치시킬 때 데이터 구조에도 주의를 기울여야 함
  • 원천 시스템의 기능적 종속성과 제약 사항들이 목표 통합 시스쳄의 것과 일치해야 함
    • 어느 시스템에서는 할인이 총 주문 금액에 적용되는 반면 다른 시스템에서는 주문을 구성하는 개별 항목에 적용
  • 데이터의 의미적 이질성과 데이터 구조는 데이터 통합 시에 여러 문제를 발생시킴
  • 여러 유형의 문제들을 신중하게 해결하여 통합 데이터의 중복과 불일치 문제를 최소화
